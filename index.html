事件 和 响应者链条

事件

UIResponse

  • UIApplication、UIViewController、UIView都继承自UIResponder，因此它们都是响应者对象，都能够接收并处理事件

  • 触摸事件

  • 加速事件

  • 远程控制事件

UIView

UIView是UIResponder的子类，可以覆盖下列4个方法处理不同的触摸事件

  • 一根或者多根手指开始触摸view，系统会自动调用view的下面方法
  • 一根或者多根手指在view上移动，系统会自动调用view的下面方法（随着手指的移动，会持续调用该方法）
  • 一根或者多根手指离开view，系统会自动调用view的下面方法
  • 触摸结束前，某个系统事件(例如电话呼入)会打断触摸过程，系统会自动调用view的下面方法

提示：touches中存放的都是UITouch对象，根据touches中UITouch的个数可以判断出是单点触摸还是多点触摸。。

UITouch

  • 当用户用一根触摸屏幕时，会创建一个与手指相关联的UITouch对象

  • 一根手指对应一个UITouch对象

UITouch的作用

  • 保存着跟手指相关的信息，比如触摸的位置、时间、阶段

  • 当手指移动时，系统会更新同一个UITouch对象，使之能够一直保存该手指在的触摸位置

  • 当手指离开屏幕时，系统会销毁相应的UITouch对象

UIEvent

  • 每产生一个事件，就会产生一个UIEvent对象

  • UIEvent：称为事件对象，记录事件产生的时刻和类型

  • 常见属性和事件类型
  • 事件产生的时间

事件的产生和传递

  • 1、发生触摸事件之后，系统会将事件添加到由一个UIApplication管理的队列中去。
  • 2、UIApplication会将队列中最前面的事件分发下去，通常，先分发事件给应用程序的主窗口(keyWindow)
  • 3、主窗口会在视图层次中找到合适的视图来处理。
  • 4、找到合适的视图控件之后，视图控件会调用touches方法处理。。

事例

  • 点击了绿色的view：
    UIApplication -> UIWindow -> 白色 -> 绿色
  • 点击了蓝色的view：
    UIApplication -> UIWindow -> 白色 -> 橙色 -> 蓝色

  • 点击了黄色的view：
    UIApplication -> UIWindow -> 白色 -> 橙色 -> 蓝色 -> 黄色

    如何找到最合适的控件来处理事件？

  • 1、自己是否能接收触摸事件？能的话往下传
  • 2、触摸点是否在自己身上？是的话到3
  • 3、从后往前遍历子控件，重复前面的两个步骤
  • 4、如果没有符合条件的子控件，那么就自己最适合处理

UIView不接受事件的情况

  • 1、不接收用户交互
  • 2、隐藏
  • 3、透明

提示：UIImageView的userInteractionEnabled默认就是NO，因此UIImageView以及它的子控件默认是不能接收触摸事件的。

响应者链的传递过程

  • 1、 将事件对象由上往下传递（父视图传到子视图），找到合适的控件(能接收事件)处理事件。
  • 2、调用touches方法。
  • 3、如果最合适控件没有touches方法，会默认调用[super touches..]方法。或者有touches方法，而且在touchees方法调用了[super touches]方法的话，就会沿着响应链往上传递。
  • 4、接着就会调用上一层的touches方法。

什么是响应链条

  • 1、相应链条是由多个响应者对象链接起来的链条。（什么是相应者对象：能处理相应的对象，继承子UIResponse）。
  • 2、利用响应链者能让多个控件处理同一个事件。

谁是上一个响应者呢？

  • 1、如果当前view是contrller的view，那么上一个响应者是controller。
  • 2、如果当前view不是contrller的view，那么上一个响应者是父视图。

例子

控制器上有一个red的控件，red上有个blue的控件。

当点击blue时：先判断blue控件有没有重写touche方法，如果没有就默认调用[super touches]方法，传递到它的父类，也就是red，red也判断是有没有touches方法，有就相应，没有就传给上一个响应者controller，一层层往上传。。。

HTTP状态码

定义：客户端访问服务器的时候返回的结果。

类别

状态码 类别 原因
1XX Information(信息性状态码) 接受的请求在处理
2XX Success(成功) 请求数据成功处理
3XX Redirection(重定向状态码) 需要添加附加操作来完成任务
4XX Client Error(客户端错误) 服务器端无法处理请求
5XX Service Error(服务器端错误) 服务器处理请求出错

常用的状态码

2XX 请求成功

  • 200 OK 表示客户端发来的请求在服务器被正常处理了.
  • 204 No Content 服务器接收的请求已经成功处理,但返回的响应报文中不含实体的主体部分.
  • 206 Partial Content 客户端进行了范围请求,服务器成功执行了这部分的GET请求.

3XX 重定向错误

  • 301 Moved Permanently 永久性重定向
  • 302 Found 临时性重定向
  • 303 See Other 请求对应的资源存在着另一个URI,应使用GET方法重新获取请求的资源.
  • 304 Not Modify 客户端发送附加条件的请求时,服务器端允许请求访问资源.
  • 307 Temporary Redirect 临时重定向

4XX 客户端错误

  • 400 Bad Request 请求报文中存在语法错误.
  • 401 Unauthorized 权鉴失败,需要有HTTP认证的认证信息.
  • 403 Forbidden 请求的资源被服务器拒绝了
  • 404 Not Fount 服务器无法找到该资源.

5XX 服务器错误

  • 500 Internal Server Error 服务器执行请求时发生错误
  • 503 Service Unavailable 服务器暂时处于超负载或正在进行停机维护,现在无法处理请求.
  • 504 (Gateway Timeout/网关超时)

初识MVVM

  • MVC :有人称之为 Massive View Controller（重量级视图控制器），也叫Model View Controller。c是为了将M和V分离的。按理说C不应该处理太多的逻辑的东西，只是让他负责管理视图的调整，当时往往很多时候，C不得不处理一些逻辑，比如网路请求数据？。。。

  • MVVM: Model-View-ViewModel ，只是MVC的一个优化版。

  • MVC - > MVVM : 我们平时在MVC模式下开发的时候，经常会有很多逻辑放在ViewController里面，这样使得Controller会特别的臃肿，有个解决方案就是讲逻辑部分提炼出来。用MVVM的术语来说就是将Model的数据转化为View可以呈现的东西的事情，所以就叫ViewModel。

MVVM的好处

  • MVVM 可以兼容你当下使用的 MVC 架构。
  • MVVM 增加你的应用的可测试性。
  • MVVM减轻Controller的重量。
