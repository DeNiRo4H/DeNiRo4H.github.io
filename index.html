Quartz 2D是一个二维绘图引擎，同时支持iOS和Mac系统 , Quartz2D的API来自于Core Graphics框架，是纯C语言的。

Quartz 2D能做什么：

  • 绘制图形 : 线条\三角形\矩形\圆\弧等
  • 绘制文字
  • 绘制\生成图片(图像)
  • 读取\生成PDF
  • 截图\裁剪图片
  • 自定义UI控件

Quartz 2D 在开发中的价值

iOS 提供的UIKit 框架，里面大部分控件都是由Quartz 2D 绘制出来的：

  • UILabel：显示文字
  • UIImageView：显示图片
  • UIButton：同时显示图片和文字（能点击）

但是如果我们要绘制复杂的控件，可能没法使用 iOS 自带的这些控件拼接成了，这样就需要自定义UI 控件。

SDImageView的实现原理

具有缓存管理，异步下载、同一URL下载次数的控制和优化等。这个类提供了一个重要的分类，UIImageView的分类，用来支持下载网络图片。

用来解决什么问题：

  • 一、在UITableView中加载大图时卡顿的问题。

    如果你加载图片的时候是在主线程中的话，那么这个操作就会阻塞UITableView的滑动。所以一般情况下会用到SDWebImage 进行异步的下载图片，异步缓存图片。

流程大致如下：

  • 1、在下载还没有完成的时候先显示占位图片placeholderImage
  • 2、将任务交给SDWebImageManager处理。
  • 3、SDWebImageManager利用SDImageCache 从缓存中查找图片是否已经下载，如果有就到SDWebImageManager，结束。
  • 4、先从内存中查找是否有缓存图片，如果没有第五步。

  • 5、如果内存缓存中没有，生成NSInvocationOperation添加到队列开始从硬盘查找图片是否已经缓存。

  • 6、根据URLKey在硬盘缓存目录下尝试读取图片文件。

  • 7、如果从硬盘缓存目录读取不到图片，说明所有缓存都不存在该图片，需要下载图片，
  • 8、共享或重新生成一个下载器SDWebImageDownloader开始下载图片。

  • 9、图片下载由NSURLConnection来做，实现相关代理来判断图片下载中、下载完成和下载失败。

    代理方法中做一些事情：

    • connection:didReceiveData:中利用ImageIO做了按图片下载进度加载效果。
    • connectionDidFinishLoading:数据下载完成后交给SDWebImageDecoder做图片解码处理。图片解码处理在一个NSOperationQueue完成，不会拖慢主线程UI。

  • 10、下载完成后将图片保存到SDImageCache中，内存缓存和硬盘缓存同时保存。

DWI也提供了UIButton+WebCache和MKAnnotationView+WebCache，方便使用。

static 、 const 和 宏

@(iOS)

苹果不推荐大量使用宏定义，推荐使用const常量。但宏有个好处就是能定义函数。

static

修饰局部变量 : 

> - 1、局部变量只初始化一次，只占一块内存。
> - 2、不改变作用域，只是改变了周期，在方法结束时不会销毁。

修饰全局变量：

  • 1、全局变量的作用域仅限于本文件

const

被修饰的变量不能在后面被修改。

  • const修饰的 *p

    被const修饰的*p 只能被赋值一次，以后不能赋值，否则编译器报错

  • const修饰的p

    被const修饰的p只能存一次地址，以后再也不能其它存地址了，否则编译器报错

  • const在声明字符串的用法

static和const联合使用

  • static将一个全局变量变成局部变量
  • const将一个局部变量变成局部常量

static const 与 宏

相同点

  • 都不能再被修改
  • 一处修改，其它都改了

不同点:

  • static const修饰变量只有一份内存
  • 宏定义，只是简单的替换，每次使用都需要创建一份内存
  • const是在预编译，而宏是编译阶段。
  • 宏不会报编译错误，const 会报编译错误。

苹果建议尽量使用const。

总结：

使用static const修饰更加高效，在同一个文件内可以使用static const取代#define

cocoaPods的安装 和 一些常见问题

一、什么是CocoaPods

CocoaPods是iOS项目的依赖管理工具，该项目源码在Github上管理。开发iOS项目不可避免地要使用第三方开源库，CocoaPods的出现使得我们可以节省设置和第三方开源库的时间。在使用CocoaPods之前，开发项目需要用到第三方开源库的时候，我们需要

  • 1.把开源库的源代码复制到项目中
  • 2.添加一些依赖框架和动态库
  • 3.设置-ObjC，-fno-objc-arc等参数
  • 4.管理他们的更新

在使用CocoaPods后，我们只需要把用到的开源库放到一个名为Podfile的文件中，然后执行pod install, Cocoapods就会自动将这些第三方开源库的源码下载下来，并且为我们的工程设置好响应的系统依赖和编译参数。

二、CocoaPods的原理

CocoaPods的原理是将所有的依赖库都放到另一个名为Pods的项目中，然后让主项目依赖Pods项目，这样，源码管理工作都从主项目移到了Pods项目中。Pods项目最终会编译成一个名为libPods.a的文件，主项目只需要依赖这个.a文件即可。

Podfile.lock：该文件用来保存安装的Pods依赖库的版本，当执行完pod install命令后，lock文件会记录当前导入第三方库的版本号及其依赖库的版本号，当再次pod install时，cocoaPods会去取lock文件记录的版本号，而不会去取最新的版本，这对于团队开发很有用。
当执行pod install之后，除了Podfile，还会生成一个名为Podfile.lock的文件，它会锁定当前各依赖库的版本，之后即使多次执行pod install也不会更改版本，只有执行pod update才会改变Podfile.lock.在多人协作的时候，这样可以防止第三方库升级时候造成大家各自的第三方库版本不一致。

iOS中事件和响应者链条

事件

UIResponse

  • UIApplication、UIViewController、UIView都继承自UIResponder，因此它们都是响应者对象，都能够接收并处理事件

  • 触摸事件

  • 加速事件

  • 远程控制事件

UIView

UIView是UIResponder的子类，可以覆盖下列4个方法处理不同的触摸事件

  • 一根或者多根手指开始触摸view，系统会自动调用view的下面方法
  • 一根或者多根手指在view上移动，系统会自动调用view的下面方法（随着手指的移动，会持续调用该方法）
  • 一根或者多根手指离开view，系统会自动调用view的下面方法
  • 触摸结束前，某个系统事件(例如电话呼入)会打断触摸过程，系统会自动调用view的下面方法

提示：touches中存放的都是UITouch对象，根据touches中UITouch的个数可以判断出是单点触摸还是多点触摸。。

UITouch

  • 当用户用一根触摸屏幕时，会创建一个与手指相关联的UITouch对象

  • 一根手指对应一个UITouch对象

UITouch的作用

  • 保存着跟手指相关的信息，比如触摸的位置、时间、阶段

  • 当手指移动时，系统会更新同一个UITouch对象，使之能够一直保存该手指在的触摸位置

  • 当手指离开屏幕时，系统会销毁相应的UITouch对象

UIEvent

  • 每产生一个事件，就会产生一个UIEvent对象

  • UIEvent：称为事件对象，记录事件产生的时刻和类型

  • 常见属性和事件类型
  • 事件产生的时间

事件的产生和传递

  • 1、发生触摸事件之后，系统会将事件添加到由一个UIApplication管理的队列中去。
  • 2、UIApplication会将队列中最前面的事件分发下去，通常，先分发事件给应用程序的主窗口(keyWindow)
  • 3、主窗口会在视图层次中找到合适的视图来处理。
  • 4、找到合适的视图控件之后，视图控件会调用touches方法处理。。

事例

  • 点击了绿色的view：
    UIApplication -> UIWindow -> 白色 -> 绿色
  • 点击了蓝色的view：
    UIApplication -> UIWindow -> 白色 -> 橙色 -> 蓝色

  • 点击了黄色的view：
    UIApplication -> UIWindow -> 白色 -> 橙色 -> 蓝色 -> 黄色

    如何找到最合适的控件来处理事件？

  • 1、自己是否能接收触摸事件？能的话往下传
  • 2、触摸点是否在自己身上？是的话到3
  • 3、从后往前遍历子控件，重复前面的两个步骤
  • 4、如果没有符合条件的子控件，那么就自己最适合处理

UIView不接受事件的情况

  • 1、不接收用户交互
  • 2、隐藏
  • 3、透明

提示：UIImageView的userInteractionEnabled默认就是NO，因此UIImageView以及它的子控件默认是不能接收触摸事件的。

响应链

下面是苹果官方给的响应者链的图：

响应者链的传递过程

  • 1、 将事件对象由上往下传递（父视图传到子视图），找到合适的控件(能接收事件)处理事件。
  • 2、调用touches方法。
  • 3、如果最合适控件没有touches方法，会默认调用[super touches..]方法。或者有touches方法，而且在touchees方法调用了[super touches]方法的话，就会沿着响应链往上传递。
  • 4、接着就会调用上一层的touches方法。

什么是响应链条

  • 1、相应链条是由多个响应者对象链接起来的链条。（什么是相应者对象：能处理相应的对象，继承子UIResponse）。
  • 2、利用响应链者能让多个控件处理同一个事件。

谁是上一个响应者呢？

  • 1、如果当前view是contrller的view，那么上一个响应者是controller。
  • 2、如果当前view不是contrller的view，那么上一个响应者是父视图。

例子

控制器上有一个red的控件，red上有个blue的控件。

当点击blue时：先判断blue控件有没有重写touche方法，如果没有就默认调用[super touches]方法，传递到它的父类，也就是red，red也判断是有没有touches方法，有就相应，没有就传给上一个响应者controller，一层层往上传。。。

HTTP状态码

定义：客户端访问服务器的时候返回的结果。

类别

状态码 类别 原因
1XX Information(信息性状态码) 接受的请求在处理
2XX Success(成功) 请求数据成功处理
3XX Redirection(重定向状态码) 需要添加附加操作来完成任务
4XX Client Error(客户端错误) 服务器端无法处理请求
5XX Service Error(服务器端错误) 服务器处理请求出错

常用的状态码

2XX 请求成功

  • 200 OK 表示客户端发来的请求在服务器被正常处理了.
  • 204 No Content 服务器接收的请求已经成功处理,但返回的响应报文中不含实体的主体部分.
  • 206 Partial Content 客户端进行了范围请求,服务器成功执行了这部分的GET请求.

3XX 重定向错误

  • 301 Moved Permanently 永久性重定向
  • 302 Found 临时性重定向
  • 303 See Other 请求对应的资源存在着另一个URI,应使用GET方法重新获取请求的资源.
  • 304 Not Modify 客户端发送附加条件的请求时,服务器端允许请求访问资源.
  • 307 Temporary Redirect 临时重定向

4XX 客户端错误

  • 400 Bad Request 请求报文中存在语法错误.
  • 401 Unauthorized 权鉴失败,需要有HTTP认证的认证信息.
  • 403 Forbidden 请求的资源被服务器拒绝了
  • 404 Not Fount 服务器无法找到该资源.

5XX 服务器错误

  • 500 Internal Server Error 服务器执行请求时发生错误
  • 503 Service Unavailable 服务器暂时处于超负载或正在进行停机维护,现在无法处理请求.
  • 504 (Gateway Timeout/网关超时)

初识MVVM

  • MVC :有人称之为 Massive View Controller（重量级视图控制器），也叫Model View Controller。c是为了将M和V分离的。按理说C不应该处理太多的逻辑的东西，只是让他负责管理视图的调整，当时往往很多时候，C不得不处理一些逻辑，比如网路请求数据？。。。

  • MVVM: Model-View-ViewModel ，只是MVC的一个优化版。

  • MVC - > MVVM : 我们平时在MVC模式下开发的时候，经常会有很多逻辑放在ViewController里面，这样使得Controller会特别的臃肿，有个解决方案就是讲逻辑部分提炼出来。用MVVM的术语来说就是将Model的数据转化为View可以呈现的东西的事情，所以就叫ViewModel。

MVVM的好处

  • MVVM 可以兼容你当下使用的 MVC 架构。
  • MVVM 增加你的应用的可测试性。
  • MVVM减轻Controller的重量。
